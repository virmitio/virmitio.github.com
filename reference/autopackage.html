Autopackage

Version:1.0

Purpose

Autopackage creates CoApp packages while trying to keep the publisher from understanding all the fine-details of the internals of the packages themselves.

Command Line Help

Autopackage uses CoApp's common command line conventions as the basis for command line options.

CoApp Autopackage utility
-------------------------------------------------------------------------------
Usage:
-------

Autopackage [options] <file.autopkg ...>

    <file.autopkg> - one or more property sheet files for composing a package

    --help                      this help

Autopackage File Format (.autopkg)

Autopackage uses CoApp's common property sheet as the basis for the .autopkg format.

Standard Rules

package

The package rule contains the manditory data for creating a package.

package { 
    name : string; 
    version : four-digit-version;
    arch : architecture;
    locations: { 
        url ...
    };
    feeds : { 
        url ...
    };
    publisher : identity-reference;
    display-name: string;
};

Package Property Descriptions

Property NameProperty Description
nameThe full name of the package, without any version, platform or public key information. (ie, "acme.widgets")
versionthe four-digit-version of the package. (ie, 1.2.3.4)
archthe binary architecture of the contents of the package. It is not permitted to have multiple binary architectures in the same package.
locations
or location		One or more URLs that will return the package file. The URL must not be a referencing webpage, as this is the URL that the package manager will attempt to fetch the package from directly.
feed
or feed		One or more URLs of feed locations that may be consulted when looking for a package upgrade.
publisheran identity-reference that contains the name, email and web address of the person or organization that created this package. This is not necessarily the author, of which should be listed in the metadata/contributors property.
display-nameA cosmetic name for the product being packaged (ie, "Acme Widgets" instead of the "acme.widgets" name property)

metadata

The metadata rule contains additional cosmetic data about the contents of the package.

metadata { 
    description : string; 
    summary : string; 
    icon : file-path
    licenses: { 
        license-reference...
    };
    author-version : string; 
    bug-tracker : url;
    publish-date : date;
    nsfw: boolean;
    stability: stability;
    tags : { 
        string...
    };
    contributors: { 
        identity-reference...
    };
};

Metadata Property Descriptions

Property NameProperty Description
descriptionA full-text description of the software package. Limited HTML may be rendered by the client.
summaryA short description of the package.
iconThe path to an image that can be used as an icon for the package. Must be limited to 256x256. PNG images will preserve alpha channels
licensesOne or more license-references that specify the license for the package.
author-versiona cosmetic string that contains the version the author has assigned to the product. (ie, this can be "1.0f" instead of in the four-digit-version "1.0.9.0"
bug-trackerA URL pointing to the location of a given product's Bug Tracker.
publish-dateThe date this package is published. Defaults to the current date/time.
nsfwa boolean flag that the publisher can set to indicate that the package is "not safe for work". NSFW packages that don't set this will incur my wrath.
stabilityan integer value ranging from -100 (very unstable) to 100 (very stable) to indicate to consumers the relative stability of the contents of the package.
tagsone or more strings containing arbitrary 'tags' about the product (ie, tools )
contributorsone or more identity-references specifying the contact information of the people who contributed to the package. It is common courtesy to list the original authors of a given open source package.

requires

CoApp 1.0-RC Currently, dependencies only match on packages, feature matching is not yet available.

The requires rule allows the publisher to specify what packages are necessary to install this package.

requires  {
    package: canonical-package-name;

    // exact matches are done in RC 
    package: exact = canonical-package-name;

    // features are targeted for RC
    feature: {
        feature-name = feature-version-modifier...
    };
}

Requires Property Descriptions

Property NameProperty Description
packageGiven a canonical package name, this specifies an explicit dependency on another package. The package-manager may associate the latest binary-compatable version instead of the specified version
packageCoApp 1.0-RC Given a canonical package name, this specifies an explicit dependency on another package, matching the version exactly.
featureCoApp 1.0-RC specifies a dependency on given feature, with an optional version, or version-plus-modifier.

compatability-policy

The compatability-policy rule defines the previous versions of this software which this version replaces. This asserts that this version is binary compatable with all replaced versions.

compatability-policy { 
    minimum: four-digit-version;
    maximum: four-digit-version;
    versions : {
        two-digit-version ...
    }; // if not specified, find the versions by looking at the feeds and finding all the major/minor versions in range.
};

Compatability Property Descriptions

Property NameProperty Description
minimumThe lowest four-digit-version for which this package is a binary-compatable replacement.
maximumThe highest four-digit-version for which this package is a binary-compatable replacement.
versionsa collection of two-digit-versions between the minimum and maximum versions that have been published

files

The files rule allows the publisher to specify groups of files that can be used in other properties.

files[reference-declaration] {
    root: folder-path;;

    include: { 
       file-mask...
    };

    trim-path: trim-path;

    destination: folder-path;
};

package-composition

package-composition[reference-declaration] {

    file-rewrite: { 
        // <list of files that get 'rewites' (ie, macro resolution)>
        config-files => "rw_${each.Name}"
        // config-files => "${each.NameWithoutExtension}";
        config-files => "${appdir}\config\${each.NameWithoutExtension}";
    };

    file-copy: {  // without the macro-handling
        config-files => "${appdir}\config\${each.Name}";
    };

    symlinks : {
        exes => @"${global-bin}\${each.name}" = @"${appdir}\${each.name}";
        scripts => @"${global-bin}\${each.name}" = @"${appdir}\${each.name}";
    }    

    symlinks : {
        powershellscripts => @"${global-powershell}\${each.name}" = @"${appdir}\${each.name}";
    }    

    environment-variables: {
        "RUBYPATH" = "${appdir}";
    };

    registry: { 
        @"${ROOTKEY}#InstallLocation" = "${APPDIR}";
    };
}

Rule/Property types

Primitive Types

string type

The string type is any literal string value..

date type

Dates are specified in ... ?

TODO: Find out what date format we use

boolean type

The boolean type is either true or false .. specifiying anything else may lead to unpredicable results.

stability-value type

The stability type is an integer ranging from -100 to 100. Negative values are considered less stable, positive values, more stable.

TODO: maybe set some guidelines for how much is how much

url type

The URL is any legal url that points to an actual location. Accepted schemes are:

SchemeDescription
httpHTTP Protocol
httpsHTTPS (secure) Protocol
filelocal file references, including UNC references.

email-address type

An email address for a given individual. (i.e. bob@somewhere.com .)

canonical-package-name type

A package reference in the format: name-version-architecutre-publickeytoken

Example:
coapp.toolkit.1.1.2.1296-any-1231123112311231

Version Types

two-digit-version class

Two 16-bit positive integers seperated by a single period. (ie, 1.0 , 4.76 ).

The numbers can range from 0 to 65535.

This is only used in compatability-policy.

four-digit-version class

Four 16-bit positive integers seperated by a periods. (ie, 1.0.0.0 , 4.76.1231.99 ).

The numbers can range from 0 to 65535.

Enumeration Types

architecture enumeration

One of x86 , x64 , any .

trim-path enumeration

One of none , all , minimal .

File/Folder Types

file-mask class
file-path class
